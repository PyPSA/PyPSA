# AGENTS.md

Contribution rules and conventions for PyPSA, for humans and AI agents alike.

## AI-assisted contributions

You're welcome to help author code, pull requests and issues. But the
conversation around a change — PR and issue descriptions, comments and review
discussions — is how maintainers and contributors understand each other, so it
must stay human and honest. Three rules:

1. **AI-generated content must be marked**, preferably with a GitHub note alert
   at the top of the generated section. Put verbose additional context (logs, generated analysis, long
   reproductions) in a collapsed `<details>` block so it doesn't drown the
   discussion.

   Don't silently mix hand-written and generated prose.

2. **The human writes their own intent.** If a pull request or issue is more
   than a self-documenting bug report, the author must write — or at least
   rewrite in their own words — their personal intent and motivation by hand,
   with everything else placed below it. Don't submit an AI-drafted description
   as your own; the maintainers want the human's voice, concise and to the
   point.

   ```markdown
   Thats why this is important for me (handwritten)

   > [!NOTE]
   > The following content was generated by AI.

   What was implemented and how

   <details>
   <summary>Full benchmark output</summary>

   ...verbose content here...

   </details>
   ```

3. **Conversations are not held through an agent.** Posting bare information via
   an agent — e.g. a log, a benchmark result, a reproduction — is fine as long
   as it's marked (see rule 1). But discussion itself — replies, answers to
   questions, review back-and-forth — must be written by the human. Don't let an
   agent argue, agree or decide on your behalf.

See the [AI-based Contributions](https://docs.pypsa.org/latest/contributing/contributing/#ai-based-contributions)
section of the documentation for the authoritative policy.

## Development workflow

- Install PyPSA with its development dependencies in editable mode:
  `pip install -e '.[dev]'`.
- Set up the linter and formatter with `pre-commit install`; `pre-commit` runs
  [`ruff`](https://docs.astral.sh/ruff) (lint and format) and other checks on
  every commit. You can also run them directly with `ruff check . --fix` and
  `ruff format .`.
- Type-check with `mypy` and run the test suite with `pytest`.
- For the authoritative, detailed instructions (extras, coverage, style and
  testing) see the [Contributing](https://docs.pypsa.org/latest/contributing/contributing/)
  documentation.

## Project conventions

- Branch off `master` for every change and open pull requests via the GitHub CLI
  (`gh`).
- Write tests for new features and bug fixes under `test/` as `test_*.py`. Run
  the tests after making changes and make sure they pass.

## Architecture in one paragraph

PyPSA (Python for Power System Analysis) models energy systems as a `Network`
that holds a set of components — buses, generators, lines, links, storage units,
loads and more — as pandas DataFrames, with time-varying data in matching
`_t` DataFrames indexed by snapshots. Optimisation builds a
[linopy](https://github.com/PyPSA/linopy) model from these components and solves
linear or mixed-integer power flow and investment problems; there are also
non-linear and linear power flow solvers. Keep new features consistent with this
component-and-DataFrame design and the linopy-based optimisation layer.
